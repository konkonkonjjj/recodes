## 1. 项目概览

减脂营养计划是一个全栈减脂饮食管理应用，支持：

- 用户身体数据录入 → 自动计算减脂计划
- 每日饮食记录 → 实时追踪热量/宏量营养素摄入
- 食物知识库浏览 → 内置 74 种食物 + 自定义录入
- **双存储架构**：MySQL（本机） + LocalStorage（手机降级）

### 3.2 模块职责详解

#### `api-client.js` — API 通信层

```javascript
// 封装所有 HTTP 请求，统一错误处理
const ApiClient = (() => {
  const BASE_URL = 'http://localhost:3001';

  async function request(method, path, body = null) {
    const opts = {
      method,
      headers: { 'Content-Type': 'application/json' },
    };
    if (body) opts.body = JSON.stringify(body);

    const res = await fetch(BASE_URL + path, opts);
    if (!res.ok) {
      const err = await res.json().catch(() => ({}));
      throw new Error(err.error || `HTTP ${res.status}`);
    }
    return res.json();
  }
  // ... 封装 getProfile / saveProfile / getMeals / addMeal / deleteMeal 等
})();
```

**设计要点**：
- `request()` 是所有请求的**唯一入口**，统一处理 JSON 序列化、错误转换
- 每个方法返回 Promise，由上层 `storage.js` 决定调用/降级策略
- `BASE_URL` 写死为 `localhost:3001`，因 MySQL 仅在本机可用

#### `storage.js` — 智能数据层（核心模块）

```javascript
const Storage = (() => {
  // 每条操作都是：先试 API，失败则 LocalStorage

  async function saveProfile(profile) {
    try {
      await ApiClient.saveProfile(profile);  // 1. 优先走 MySQL
    } catch (e) {
      _save(KEYS.PROFILE, profile);          // 2. 失败 → LocalStorage
    }
  }

  async function loadProfile() {
    try {
      const data = await ApiClient.getProfile();  // 1. 优先从 MySQL 读
      if (data) return transformProfile(data);
    } catch (e) { /* 静默降级 */ }
    return _load(KEYS.PROFILE, null);            // 2. 失败 → LocalStorage
  }
})();
```

**降级触发场景**：
| 场景 | 行为 |
|------|------|
| 本机 `localhost:3001` + 后端运行中 | → MySQL |
| 本机 + 后端未启动 | → LocalStorage |
| 手机 GitHub Pages（无后端） | → LocalStorage |
| 网络异常/后端崩溃 | → LocalStorage |

**关键设计**：降级是**静默**的，前端 UI 代码完全感知不到存储介质切换。

#### `calculator.js` — 计算引擎

纯函数模块，所有计算不依赖任何外部数据：

| 函数 | 公式 | 输出 |
|------|------|------|
| `calcBMR()` | Mifflin-St Jeor | BMR (kcal/天) |
| `calcTDEE()` | BMR × 活动系数 | TDEE (kcal/天) |
| `calcTarget()` | TDEE - 缺口 | 目标热量 (≥ 1200♀ / 1500♂) |
| `calcMacros()` | 热量 × 比例 ÷ 密度 | 蛋白质/碳水/脂肪 (g) |
| `calcFoodNutrition()` | 每100g数据 × 克数/100 | 实际摄入营养 |
| `calcRemaining()` | 目标 - 已摄入 | 剩余可摄入量 |
| `generateMealPlan()` | 热量 × 餐次比例 | 每餐分配建议 |

**Mifflin-St Jeor 公式**：
```
男性: BMR = 10×体重(kg) + 6.25×身高(cm) - 5×年龄 + 5
女性: BMR = 10×体重(kg) + 6.25×身高(cm) - 5×年龄 - 161
```

**活动系数**：
| 级别 | 系数 | 说明 |
|------|------|------|
| sedentary | 1.200 | 久坐 |
| light | 1.375 | 1-3天/周 |
| moderate | 1.550 | 3-5天/周 |
| active | 1.725 | 6-7天/周 |

**宏量分配**：
| 营养素 | 热量占比 | 热量密度 | 计算公式 |
|--------|----------|----------|----------|
| 蛋白质 | 30% | 4 kcal/g | `目标热量 × 0.30 ÷ 4` |
| 碳水化合物 | 45% | 4 kcal/g | `目标热量 × 0.45 ÷ 4` |
| 脂肪 | 25% | 9 kcal/g | `目标热量 × 0.25 ÷ 9` |

#### `food-db.js` — 食物知识库

```javascript
const FoodDB = (() => {
```

**7 大分类**：主食 / 肉类蛋豆 / 蔬菜 / 水果 / 乳制品 / 零食糕点 / 饮品

#### `meal-log.js` — 饮食记录管理

```javascript
const MealLog = (() => {
  async function addFood(date, mealType, food, grams) {
    const nutrition = Calculator.calcFoodNutrition(food, grams);

    // 1. 优先通过 API 添加（逐条写入 MySQL）
    try {
      return await ApiClient.addMeal(date, mealType, food.name, food.category, grams, nutrition);
    } catch (e) {
      // 2. 降级：整份日志保存到 LocalStorage
      const log = await load(date);
      log.meals[mealType].push({ id: generateId(), foodName: food.name, ... });
      await save(date, log);
    }
  }

  // 同步方法：直接操作内存中的日志对象，避免不必要的异步等待
  function calcConsumedFromLog(log) { /* 纯计算，无 I/O */ }

  // 异步方法：从存储加载日志后再计算
  async function calcConsumed(date) {
    const log = await load(date);
    return calcConsumedFromLog(log);
  }
})();
```

**设计要点**：
- `addFood` / `removeFood` 是**逐条操作**，API 路径也是一条一增删
- 提供**同步版本** `calcConsumedFromLog()`，UI 渲染时直接用缓存对象，避免重复异步加载
- 降级时整体序列化到 LocalStorage（一个日期一个 key）

## 4. 后端架构

### 4.1 Express 服务入口 (`server/index.js`)

```javascript
const express = require('express');
const cors = require('cors');

const app = express();
app.use(cors());                    // 允许跨域（前端可能在不同端口）
app.use(express.json());           // 解析 JSON 请求体
app.use(express.static('..'));     // 托管前端静态文件

// API 路由
app.use('/api/profile', require('./routes/profile'));
app.use('/api/meals',   require('./routes/meals'));
app.use('/api/foods',   require('./routes/foods'));

app.listen(3001);
```

### 4.2 MySQL 连接池 (`server/db.js`)

```javascript
const pool = mysql.createPool({
  host: 'localhost',
  user: 'root',
  password: '123456',
  database: 'bmr_app',
  connectionLimit: 10,       // 最大 10 个并发连接
  charset: 'utf8mb4',        // 支持中文和 Emoji
});
```

**为什么用连接池而非单连接**：
- 避免每次请求都重新建立 TCP 连接
- 限制最大并发数保护 MySQL 不被打爆
- 自动处理连接断开重连

### 4.3 路由实现详解

#### `routes/profile.js` — 用户信息

```
GET  /api/profile      → SELECT * FROM user_profile WHERE id=1
POST /api/profile      → INSERT ... ON DUPLICATE KEY UPDATE
```

**关键 SQL**：
```sql
INSERT INTO user_profile (id, gender, age, height, weight, activity_level, deficit)
VALUES (1, ?, ?, ?, ?, ?, ?)
ON DUPLICATE KEY UPDATE
  gender = VALUES(gender),
  age = VALUES(age),
  ...
```

`ON DUPLICATE KEY UPDATE` 实现**存在则更新，不存在则插入**，保证表中始终只有一行（id=1）。

#### `routes/meals.js` — 饮食记录

```
GET    /api/meals/:date
                → SELECT * FROM meal_records WHERE record_date = ?
                → 按 meal_type 分组返回 { meals: { breakfast:[], lunch:[], ... } }

POST   /api/meals/:date
                → INSERT INTO meal_records (record_date, meal_type, food_name, ...)

DELETE /api/meals/:date/:mealType/:id
                → DELETE FROM meal_records WHERE record_date=? AND meal_type=? AND id=?
```

**GET 返回格式**（与前端数据结构对齐）：
```json
{
  "meals": {
    "breakfast": [
      { "id": "1", "foodName": "鸡蛋", "grams": 100,
        "nutrition": { "calories": 155, "protein": 12.6, "carbs": 1.1, "fat": 10.6 } }
    ],
    "lunch": [],
    "dinner": [],
    "snack": []
  }
}
```

#### `routes/foods.js` — 自定义食物

```
GET    /api/foods/custom    → SELECT * FROM custom_foods
POST   /api/foods/custom    → INSERT (含重复名检查, 409 冲突)
DELETE /api/foods/custom/:id → DELETE
```

**重复名检查**：
```javascript
const [existing] = await pool.query('SELECT id FROM custom_foods WHERE name = ?', [name]);
if (existing.length > 0) {
  return res.status(409).json({ error: '食物名称已存在' });
}
```

## 5. 数据库设计

### 5.1 库表结构

```
bmr_app (utf8mb4)
├── user_profile      — 用户信息（单行）
├── meal_records      — 饮食记录（按日期索引）
└── custom_foods      — 自定义食物
```

### 5.2 `user_profile` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PRIMARY KEY DEFAULT 1 | 固定为 1，保证单行 |
| gender | VARCHAR(10) | `male` / `female` |
| age | INT | 年龄 |
| height | FLOAT | 身高 (cm) |
| weight | FLOAT | 体重 (kg) |
| activity_level | VARCHAR(20) | 活动水平枚举值 |
| deficit | INT DEFAULT 500 | 热量缺口 (kcal) |
| updated_at | TIMESTAMP | 自动更新修改时间 |

**设计理由**：`id` 固定为 1 + `ON DUPLICATE KEY UPDATE`，确保表中始终只有一行用户数据，无需应用层判断 INSERT vs UPDATE。

### 5.3 `meal_records` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT AUTO_INCREMENT PK | 自增主键 |
| record_date | DATE NOT NULL | 记录日期 |
| meal_type | ENUM('breakfast','lunch','dinner','snack') | 餐次 |
| food_name | VARCHAR(100) | 食物名称 |
| food_category | VARCHAR(20) | 分类 |
| grams | INT | 食用克数 |
| calories | INT | 热量 (kcal) |
| protein | FLOAT | 蛋白质 (g) |
| carbs | FLOAT | 碳水 (g) |
| fat | FLOAT | 脂肪 (g) |
| created_at | TIMESTAMP | 创建时间 |

**索引设计**：
```sql
INDEX idx_date (record_date)                    -- 按日期查询
INDEX idx_date_meal (record_date, meal_type)    -- 按日期+餐次查询
```

**设计决策**：每条食物记录存一份营养数据（calories/protein/carbs/fat），而非通过外键关联食物表。原因：
1. 自定义食物可能被删除，但历史记录应保留当时的营养值
2. 用户可能凭经验输入食物（未来功能），不一定从库中选
3. 查询更简单，无需 JOIN

### 5.4 `custom_foods` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT AUTO_INCREMENT PK | 自增主键 |
| name | VARCHAR(100) UNIQUE | 食物名称（唯一） |
| category | VARCHAR(20) | 分类 |
| calories/protein/carbs/fat | FLOAT | 每 100g 营养成分 |
| created_at | TIMESTAMP | 创建时间 |

## 6. 核心计算引擎

### 6.1 计算流程图

```
用户输入
  ↓
calcBMR(gender, weight, height, age)
  ↓ BMR
calcTDEE(bmr, activityLevel)
  ↓ TDEE
calcTarget(tdee, deficit, gender)
  ↓ 目标热量 (下限保护)
calcMacros(targetCalories)
  ↓ { protein, carbs, fat } 克数
generateMealPlan(targetCalories, macros)
  ↓ { breakfast, lunch, dinner, snack } 每餐分配
```

### 6.2 热量缺口调节

```
减脂目标 = TDEE - 缺口

缺口范围: 300 ~ 800 kcal (滑块可调，默认 500)

下限保护:
  男性: 不低于 1500 kcal/天
  女性: 不低于 1200 kcal/天
```

**减脂原理**：1kg 脂肪 ≈ 7700 kcal。每天 500 kcal 缺口 → 每周约减 0.45 kg。

### 6.3 食物营养换算

```javascript
function calcFoodNutrition(food, grams) {
  const ratio = grams / 100;  // 食物库数据以 100g 为基准
  return {
    calories: Math.round(food.calories * ratio),
    protein: +(food.protein * ratio).toFixed(1),
    carbs:   +(food.carbs   * ratio).toFixed(1),
    fat:     +(food.fat     * ratio).toFixed(1),
  };
}
```

### 6.4 剩余热量计算

```javascript
function calcRemaining(target, targetMacros, consumed) {
  return {
    calories: target - consumed.calories,
    protein:  Math.max(0, targetMacros.protein - consumed.protein),
    carbs:    Math.max(0, targetMacros.carbs   - consumed.carbs),
    fat:      Math.max(0, targetMacros.fat     - consumed.fat),
  };
}
```

`Math.max(0, ...)` 确保剩余不会出现负数，UI 上显示"已达标 ✓"。

### 6.5 餐次分配

```
早餐 30% | 午餐 35% | 晚餐 25% | 加餐 10%
```

每餐按相同比例分配热量和宏量营养素。

```
│   → updateDashboard() ← 更新进度条                  │
└────────────────────┬───────────────────────────────┘
                     │
┌────────────────────┴───────────────────────────────┐
│ Express: routes/meals.js                           │
│   POST → INSERT INTO meal_records ...              │
│   GET  → SELECT * FROM meal_records WHERE date=?   │
└────────────────────┬───────────────────────────────┘
                     │
┌────────────────────┴───────────────────────────────┐
│ MySQL: bmr_app.meal_records                        │
└────────────────────────────────────────────────────┘
```

### 7.3 降级流程（API 不可用）

```
浏览器
  → MealLog.addFood(...)
      → ApiClient.addMeal(...)
          → fetch POST → ❌ 连接失败 (ERR_CONNECTION_REFUSED)
      → catch (e)
          → 降级到 LocalStorage:
              → load(date) from localStorage
              → log.meals['lunch'].push(entry)
              → save(date, log) to localStorage

  → renderMealLog()
      → Storage.loadDailyLog(date)
          → ApiClient.getMeals(date)
              → fetch GET → ❌ 连接失败
          → catch (e)
              → load from localStorage ✅
```

**降级后的数据隔离**：LocalStorage 降级写入的数据不会自动同步回 MySQL。下次 API 恢复时，MySQL 中没有降级期间的数据。这是有意设计 — 避免复杂的数据同步冲突。

## 9. PWA 离线方案

### 9.1 manifest.json

```json
{
  "name": "减脂营养计划",
  "short_name": "减脂计划",
  "display": "standalone",    // 独立窗口（无浏览器地址栏）
  "orientation": "portrait",  // 竖屏锁定
  "theme_color": "#16a34a",   // 状态栏颜色（绿色）
  "background_color": "#16a34a"
}
```

### 9.2 Service Worker (`sw.js`)

**缓存策略**：Cache First（缓存优先）

```
请求 → 检查缓存 → 命中 → 直接返回（离线可用）
                  → 未命中 → fetch 网络 → 成功 → 缓存 + 返回
                                         → 失败 → 降级 HTML
```

**缓存清单**：
```
/index.html
/css/style.css
/js/{api-client,storage,calculator,food-db,meal-log,app}.js
/manifest.json
/icon-192.png
/icon-512.png
```

**生命周期**：
1. `install` → 预缓存所有静态资源
2. `activate` → 清理旧版本缓存
3. `fetch` → 拦截所有 GET 请求，缓存优先

### 9.3 添加主屏幕效果

添加后，App 以**独立窗口**启动（无地址栏/工具栏），体验接近原生 App。离线时仍可打开使用（LocalStorage 降级模式）。

## 10. 部署方案

### 10.1 本地开发

```bash
# 终端 1：启动后端
cd server && node index.js

# 终端 2：启动前端
python -m http.server 8080 --bind 0.0.0.0

# 或一键启动
start.bat
```

### 10.2 手机访问（同一 WiFi）

启动服务后，手机通过 `http://电脑IP:8080` 访问。
此时 API 不可达（手机不在 localhost），自动使用 LocalStorage 降级。

### 10.3 GitHub Pages（公网永久链接）

推送代码后自动部署：`https://konkonkonjjj.github.io/recodes`

手机浏览器打开 → 「添加到主屏幕」→ 像原生 App 使用。
无后端 → 数据走 LocalStorage，功能完全正常。

### 10.4 独立离线文件

`app-standalone.html`（75KB）包含所有 CSS/JS 内联。发到微信/邮件 → 手机浏览器直接打开。零部署、零服务器。
